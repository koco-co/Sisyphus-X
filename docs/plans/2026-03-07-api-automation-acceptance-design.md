# 接口自动化模块完整性验收设计（含系统设置）- Design

**日期**: 2026-03-07  
**项目**: Sisyphus-X  
**目标项目名**: `Sisyphus-API-Test`  
**范围模块**: 接口自动化全流程 + 系统设置（全局参数、环境管理）

---

## 1. 背景与目标

本次工作目标是执行一次高压完整性验收，覆盖接口自动化主流程与系统设置关键能力，最终交付可复验的全流程操作视频（`.webm`）。

验收必须满足以下硬约束：

1. 全程使用深色模式、简体中文界面。
2. 前端操作需要覆盖所有关键交互，不仅限于按钮/输入框/下拉框/列表。
3. 验收需与后端 HTTP 接口闭环，并通过 PostgreSQL 做关键数据落库校验。
4. 当前环境历史实体视为脏数据，不可作为断言基线。
5. 主流程数据保留，删除类操作使用独立测试数据并清理。
6. 执行中若遇阻塞，先修复 Bug，再继续验收。

---

## 2. 总体执行架构

### 阶段 0：环境基线检查

- 启动并确认前端、后端、PostgreSQL 可用。
- 验证前端访问、`/api/v1` 健康、数据库连接正常。
- 仅使用仓库内配置文件给出的数据库环境参数。

### 阶段 1：测试数据与上下文准备

- 以 `Sisyphus-API-Test` 为唯一业务上下文。
- 不复用、不参考历史脏数据。
- 构建两套隔离数据集：
  - **数据集 A（保留）**：主流程全链路数据。
  - **数据集 B（可销毁）**：删除/破坏性验证专用数据。
- 所有新数据加唯一标识，确保可追溯、可复验。

### 阶段 2：前端高压完整性验收（录制视频）

- 覆盖接口自动化主链路：
  - 项目管理
  - 关键字配置
  - 接口定义与调试
  - 场景编排
  - 测试计划
  - 测试报告
- 覆盖系统设置：
  - 全局参数
  - 环境管理（含接口定义页环境管理弹窗）
- 全程录制视频并保留关键节点展示。

### 阶段 3：后端接口 + PostgreSQL 闭环校验

- 关键动作执行后，进行三层验证：
  - UI 反馈正确
  - API 返回正确
  - DB 落库状态正确
- 重点校验字段值、关联关系、状态流转、时间戳合理性。

### 阶段 4：阻塞修复机制

- 任一节点阻塞时，执行：
  - 复现
  - 根因定位
  - 最小修复
  - 局部回归
  - 主链路回归
  - 从断点继续验收
- 每个修复记录问题现象、根因、修复点、回归结果。

### 阶段 5：交付固化

- 导出单文件 `.webm`。
- 输出覆盖结果、缺陷修复记录、DB 闭环证据摘要。
- 保留数据集 A，清理数据集 B。

---

## 3. 覆盖矩阵（高压版）

### 3.1 交互元素覆盖

页面可见且可交互元素需覆盖：

- 按钮、链接
- 输入框、文本域
- 下拉框、自动补全
- 单选框、复选框、开关
- 日期/时间选择器
- 搜索框、筛选器、排序器、分页器
- 标签页、弹窗、抽屉、确认框
- 代码编辑器（如有）
- 上传/下载入口（如有）

### 3.2 状态覆盖

需覆盖并校验：

- 初始态、加载态、空态
- 成功态、失败态
- 禁用态、只读态
- 列表滚动与分页切换态
- 筛选后态、编辑回填态

### 3.3 流程覆盖

- 正向链路：创建 → 编辑 → 保存 → 执行 → 报告查看。
- 逆向链路：取消、返回、重试、幂等重复点击。
- 删除链路：仅对数据集 B 执行并清理，数据集 A 不删除。
- 跨页面往返后数据一致性校验。

### 3.4 异常与容错覆盖

- 请求超时
- 4xx / 5xx 错误
- 字段校验失败
- 重名冲突
- 依赖数据缺失
- 中断后恢复

---

## 4. 闭环判定标准（UI/API/DB）

每个关键业务动作必须满足：

1. 前端提示/状态正确；
2. 后端接口返回成功或符合预期失败；
3. PostgreSQL 对应记录存在且字段正确；
4. 关键关联关系有效（如计划-场景、执行-报告、环境-接口上下文）；
5. 状态流转完整且无异常回滚。

---

## 5. 里程碑与停止规则

### 里程碑

- **M1**：环境健康通过。
- **M2**：数据集 A 构建并首轮主链路可跑。
- **M3**：高压覆盖矩阵执行完成。
- **M4**：数据集 B 删除验证完成且已清理。
- **M5**：视频与证据完成固化。

### 停止/恢复规则

- 发现 **P0** 立即停止当前阶段并优先修复。
- 连续两次同点失败，切换为根因优先处理模式。
- 修复后必须完成局部回归 + 主链路关键回归，再继续扩展覆盖。

---

## 6. 缺陷分级与处理策略

- **P0**：主链路阻断、执行结果错误、数据一致性破坏；必须修复后继续。
- **P1**：关键功能异常但可绕行；本轮优先修复，无法修复需单列风险。
- **P2/P3**：非阻塞问题，记录并评估后续处理。

---

## 7. 最终验收出口标准

必须全部满足：

1. 接口自动化全链路从创建到报告可完整跑通；
2. 系统设置（全局参数、环境管理）可用并与主链路关联成功；
3. 高压覆盖矩阵执行完成；
4. 所有 P0 已关闭；
5. 主流程数据集 A 保留；
6. 删除测试数据集 B 清理完成；
7. 提供可播放 `.webm` 视频（深色模式 + 简体中文）；
8. 提供缺陷修复与 DB 闭环证据摘要。

---

## 8. 交付物

1. 全流程操作视频：`webm` 单文件。
2. 覆盖执行结果清单（通过/失败/阻塞点）。
3. 缺陷修复记录（现象、根因、修复、回归结果）。
4. PostgreSQL 闭环校验摘要（关键实体与状态流转）。

---

## 9. 执行注意事项

- 先保证稳定、再扩展覆盖；避免“覆盖率高但不可复现”。
- 所有新增测试数据必须可追溯；禁止复用历史脏数据作基线。
- 严格区分 A/B 数据集，避免误删主流程资产。
- 视频中需展示主链路关键节点成功状态，确保可审阅与可复验。
