## 1. 概述

### 1.1 项目背景

Sisyphus-X 是一款面向测试团队的自动化测试管理平台，提供从「接口定义 → 场景编排 → 测试计划 → 测试报告」的全链路接口自动化能力。核心执行器 `sisyphus-api-engine` 以 YAML 驱动，支持 CLI 调用和多种报告格式输出。

### 1.2 验收目标

- 验证接口自动化模块六大功能全部正常使用
- 确保每个模块有足够多的测试数据 (50+条)
- 验证所有异常数据的前后端校验
- 验证页面动画过渡流畅，组件样式全局统一
- 重点验证与 sisyphus-api-engine 的联调集成

### 1.3 验收范围

| 模块 | 优先级 | 说明 |
|------|--------|------|
| 项目管理 | P1 | 项目基础信息 + 数据库配置 |
| 关键字配置 | P1 | 内置/自定义关键字管理 |
| 接口定义 | P0 | ⚠️ 已知问题模块，需重点修复 |
| 场景编排 | P0 | 🔴 核心联调模块 |
| 测试计划 | P0 | 🔴 核心联调模块 |
| 测试报告 | P1 | 报告查看、Allure 集成 |

## 2. 测试架构设计

### 2.1 目录结构

```
tests/e2e/
├── playwright.config.ts              # Playwright 配置
├── auth.setup.ts                     # 认证 setup (开发模式跳过登录)
│
├── specs/                            # 测试用例 (按模块划分)
│   ├── 01-project.spec.ts            # 项目管理 + 数据库配置
│   ├── 02-keyword.spec.ts            # 关键字配置
│   ├── 03-interface.spec.ts          # 接口定义 ⚠️ 已知问题模块
│   ├── 04-scenario.spec.ts           # 场景编排 🔴 核心联调
│   ├── 05-plan.spec.ts               # 测试计划 🔴 核心联调
│   ├── 06-report.spec.ts             # 测试报告
│   ├── ui-ux-validation.spec.ts      # UI/UX 全局验收
│   └── validation.spec.ts            # 异常数据校验
│
├── pages/                            # Page Object Model
│   ├── project-list.page.ts
│   ├── keyword-list.page.ts
│   ├── interface-editor.page.ts
│   ├── scenario-editor.page.ts
│   ├── plan-editor.page.ts
│   └── report-list.page.ts
│
├── utils/                            # 工具函数
│   ├── data-generator.ts             # 动态测试数据生成器 (50+条)
│   ├── api-client.ts                 # 后端 API 直接调用
│   ├── engine-helper.ts              # sisyphus-api-engine 辅助
│   ├── wait-helpers.ts               # 动画等待辅助
│   └── assertion-helpers.ts          # 自定义断言 (动画、样式)
│
└── fixtures/                         # 测试 fixtures
    ├── base.fixture.ts               # 基础 fixture
    └── test-data.fixture.ts          # 测试数据 fixture
```

### 2.2 执行顺序

按依赖关系顺序执行：

```
1. 项目管理 → 创建测试项目和数据库配置
2. 关键字配置 → 验证内置关键字、创建自定义关键字
3. 接口定义 → 创建接口、调试、环境管理 (重点修复)
4. 场景编排 → 编排步骤、调试执行、引擎联调 (核心)
5. 测试计划 → 关联场景、批量执行、实时报告 (核心)
6. 测试报告 → 查看报告、Allure 集成
7. UI/UX 验收 → 动画、样式、交互规范
8. 异常校验 → 表单验证、业务规则、引擎错误
```

使用 TypeScript 编写数据生成器，在测试过程中动态创建测试数据：

```typescript
// tests/e2e/utils/data-generator.ts

export class DataGenerator {
  private static counter = 0;

  static uniqueId(): string {
    return `${Date.now()}_${++this.counter}`;
  }

  static generateProjects(count: number): ProjectData[] { /* ... */ }
  static generateDatabaseConfigs(count: number): DatabaseConfigData[] { /* ... */ }
  static generateKeywords(count: number): KeywordData[] { /* ... */ }
  static generateInterfaces(count: number): InterfaceData[] { /* ... */ }
  static generateScenarios(count: number): ScenarioData[] { /* ... */ }
  static generatePlans(count: number): PlanData[] { /* ... */ }
}
```

### 3.2 测试数据量规划

| 模块 | 数据量 | 说明 |
|------|--------|------|
| 项目管理 | 10+ 项目 | 每个项目 5+ 数据库配置 |

| 接口定义 | 50+ 接口 | 多种 HTTP 方法，含目录组织 |
| 场景编排 | 30+ 场景 | 不同步骤类型组合 |
| 测试计划 | 15+ 计划 | 不同场景组合 |
| 测试报告 | 执行产生 | 计划执行自动生成 |

## 5. UI/UX 验收标准

### 5.1 动画过渡验收

| 验收项 | 通过标准 | 测试方法 |
|--------|----------|----------|
| 页面切换 | 平均 FPS ≥ 30 | page.evaluate 测量 |
| 弹窗开关 | 300ms 内完成 | 时间测量 |
| 列表加载 | 骨架屏 → 内容流畅 | 视觉检查 |
| Toast 提示 | 进度条倒计时动画 | 视觉检查 |
| Flow 画布 | 拖拽流畅 | FPS 测量 |

### 5.2 组件样式统一性

| 组件 | 验收标准 |
|------|----------|
| 搜索框 | placeholder、边框、圆角全局一致 |
| 分页组件 | 样式全局一致 |
| 下拉框 | 选项样式、选中状态一致 |
| 暂无数据 | 图标、文案全局一致 |
| 开关按钮 | 启用/禁用状态样式一致 |
| Toast 提示 | 右上角固定、进度条倒计时 |

### 5.3 交互规范验收

| 交互 | 验收标准 |
|------|----------|
| 删除操作 | 所有删除按钮需二次确认弹窗 |
| 枚举字段筛选 | 筛选按钮 + 重置/确定 |
| 标签页机制 | 多标签页打开、关闭当前/其他/全部 |
| 查询触发 | 支持回车和按钮两种方式 |

## 6. 异常数据校验

### 6.1 表单验证

| 验证项 | 前端校验 | 后端校验 |
|--------|----------|----------|
| 必填字段为空 | 错误提示显示 | 422 响应 |
| 字段长度超限 | 错误提示显示 | 422 响应 |
| 格式错误 | 错误提示显示 | 422 响应 |

### 6.2 业务规则验证

| 规则 | 验证方式 |
|------|----------|
| 数据库配置必须测试连接 | 未测试连接 → 保存按钮禁用 |
| 场景必须关联项目 | 未选择项目 → 无法保存 |
| 计划必须包含场景 | 空场景列表 → 执行按钮禁用 |

### 6.3 引擎错误处理

| 错误码 | 验证方式 |
|--------|----------|
| YAML_PARSE_ERROR | 触发错误 → 验证友好显示 |
| REQUEST_TIMEOUT | 触发错误 → 验证错误信息 |
| ASSERTION_FAILED | 触发错误 → 验证 expected vs actual 差异 |

## 7. 引擎集成测试

### 7.1 YAML 生成验证

验证后端生成的 YAML 文件符合 [YAML 输入规范](../api-engine/Sisyphus-api-engine%20YAML%20输入规范.md)：

- config 必填字段完整 (name, project_id, scenario_id)
- teststeps 结构正确
- 变量引用语法正确 ({{variable}})
- ddts 数据驱动格式正确

### 7.2 JSON 输出验证

验证引擎返回的 JSON 符合 [JSON 输出规范](../api-engine/Sisyphus-api-engine%20JSON%20输出规范.md)：

- 顶层必填字段完整
- summary 统计正确
- steps 详情结构正确
- 错误信息格式正确

## 8. 执行计划

### 8.1 阶段划分

```
阶段 1: 环境准备 (预计 1 小时)
├── 确认后端/前端服务正常启动
├── 确认 PostgreSQL/Redis/MinIO 中间件运行
├── 确认 sisyphus-api-engine 已安装
└── 初始化测试目录结构

阶段 2: 核心模块测试 (预计 4-6 小时)
├── 01-project.spec.ts      → 项目管理 (30 min)
├── 02-keyword.spec.ts      → 关键字配置 (30 min)
├── 03-interface.spec.ts    → 接口定义 ⚠️ (60 min)
├── 04-scenario.spec.ts     → 场景编排 🔴 (90 min)
├── 05-plan.spec.ts         → 测试计划 🔴 (90 min)
└── 06-report.spec.ts       → 测试报告 (30 min)

阶段 3: UI/UX 验收 (预计 1 小时)
├── 动画过渡验收
├── 组件样式统一性
└── 交互规范验收

阶段 4: 异常校验 (预计 1 小时)
├── 表单验证
├── 业务规则验证
└── 引擎错误处理

阶段 5: 问题修复 (迭代)
├── 分析测试失败原因
├── 修复前端问题
├── 修复后端问题
├── 修复引擎问题 (如需要)
└── 重新运行测试验证
```

### 8.2 预期总时长

**8-12 小时** (含问题修复)
